# Hermes Skill：玩具玩法对话设计师 · 需求与设计 v2

> 目标：像 grill-me 一样，通过多轮对话逐步明确用户需求，最终生成可执行的 Lua 脚本 + 步骤化解读

---

## 1. 核心差异（vs v1 的 one-shot 方案）

| 维度 | v1（废弃） | v2（本版） |
|------|-----------|-----------|
| 交互方式 | 一次输入→一次输出 | **多轮对话**，逐层推进 |
| 对话驱动力 | 用户主动描述 | **Agent 主动提问**，引导用户思考 |
| 模糊需求处理 | 用默认值填充 | **追问直到明确** |
| 修改能力 | 重新描述重新生成 | **迭代调整**（"降30%""再晚点"） |
| 推荐机制 | 无 | **每次提问给推荐选项** |
| 最终产出 | Lua + 步骤 | **玩法命名** + Lua + 步骤（可多次修改） |

---

## 2. 调用方式

Skill 在 **交互式 Hermes session** 中通过 `/skill` 加载：

```
# 在 Hermes 对话中
/skill toy-play-generator

# 或者 FastAPI 包装服务创建 session 时预加载：
hermes -s toy-play-generator
```

**关键区别**：不走 `-z`（oneshot），走全交互模式。

### 2.1 玩具上下文注入

已连接玩具的信息通过**系统提示词**或**对话首条消息**注入：

```
[系统提示]
当前已连接玩具：
- vibe_1 (震动棒): rate(motor_a,motor_b,motor_c) 0-20, set_mode(motor_select,mode_id), stop()
- ems_1 (电击器二代): set_channel_fixed(channel,mode_id,intensity), set_channel_realtime(...), stop_all()
- enema_1 (灌肠机): fill(seconds), drain(seconds), pause()

运行时函数：wait(ms), print(msg)
玩具调用方式：toy.<id>:<method>(<args>)
```

Skill **不关心玩具信息怎么来的**——它只需要假设运行环境里已经有 `__CONNECTED_TOYS__` 这类上下文变量。

## 3. 对话流程设计

### 3.1 总对话树

```
         ┌─────────────────────────┐
         │  🎬 开场介绍 + 玩具概览  │
         └────────────────────┬────┘
                              │
         ┌────────────────────▼────┐
         │  Phase 1: 基础需求       │
         │  · 选玩具（默认全选）     │
         │  · 总时长                │
         │  · 整体强度曲线          │
         └────────────────────┬────┘
                              │
         ┌────────────────────▼────┐
         │  Phase 2: 节奏与感觉     │
         │  · 节奏模式              │
         │  · 感觉偏好              │
         │  · 特殊需求              │
         └────────────────────┬────┘
                              │
         ┌────────────────────▼────┐
         │  Phase 3: 玩具协调       │
         │  · 主从/并列/交替        │
         │  · 各玩具特化问题        │
         └────────────────────┬────┘
                              │
         ┌────────────────────▼────┐
         │  🎯 生成玩法              │
         │  · 展示玩法名称          │
         │  · 展示步骤解读          │
         │  · 展示 Lua 脚本         │
         └────────────────────┬────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
      ┌─────────────────┐          ┌─────────────────┐
      │  满意 → 定稿输出  │          │  不满意 → 迭代   │
      │                  │          │  "降30%强度"     │
      │  · 方案名称      │          │  "灌肠机晚点"    │
      │  · Lua 脚本      │          │  "加一个循环"    │
      │  · 步骤解读      │          │  → 重新生成       │
      └─────────────────┘          └─────────────────┘
```

### 3.2 各 Phase 详细对话设计

#### Phase 0：开场

```
Agent: "🎮 欢迎来到玩法设计模式！
我检测到你已连接了以下玩具：
  📳 震动棒 (vibe_1)
  ⚡ 电击器二代 (ems_1)
  💧 灌肠机 (enema_1)

我来一步步帮你设计一套专属玩法。
先从基础开始——你想用哪几个玩具？"

[用户回答]
```

选项：全选 / 选部分 / 先看看

#### Phase 1：问 3 个基础问题

**Q1：选择玩具**
- 推荐：全选（除非用户明确说不用哪个）

**Q2：总时长**
- 推荐：3-5 分钟
- 备选：30秒快捷 / 1分钟 / 5-10分钟沉浸 / 自定义

**Q3：整体强度曲线**
- 推荐：渐进式（从低到高逐步爬升）
- 备选：温和（全程低强度）/ 脉冲（高低交替）/ 暴烈（短时高强度）

#### Phase 2：问 2-3 个感觉问题

**Q4：节奏模式**
- 推荐：混合节奏（前期舒缓，中期起伏，后期密集冲刺）
- 备选：恒定匀速 / 脉冲间歇 / 台阶递增 / 自由发挥

**Q5：感觉偏好**（根据已连接的玩具类型动态调整问题）

```
如果有震动棒：   "震动棒的感觉偏好？酥麻/沉重/拍打/混合"
如果有电击器：   "电击器的体感偏好？轻柔针刺/肌肉震颤/强烈冲击/同频同步"
如果有灌肠机：   "灌肠机的节奏？持续注水/脉冲注水/快速注排/慢慢充盈"
```

**Q6：特殊需求**（可选，如果前面已经明确可以跳过）
- 有没有不想用的模式/感觉？
- 有没有想重点强化的部分？

#### Phase 3：问玩具协调问题（根据参与玩具数量）

**2个玩具** → 问配合方式：
- 推荐：交替错峰（A 强时 B 弱，反之亦然）
- 备选：同步同频 / 主从搭配 / 渐进叠加

**3+个玩具** → 问整体结构：
- 推荐：分阶段推进（前戏→积累→高潮→收尾）
- 备选：全部一起 / 逐个引入 / 分两路配合

**针对灌肠机**（如果参与）：
- 灌肠机的时机：刚开始就注水、中途加入、最后冲刺前

**针对电击器二代**（如果参与）：
- 固定模式还是自定义参数？固定模式几号？
- 要不要频率模式（100步脉冲序列）？

#### Phase 4：生成

```
Agent: "📋 好的，我来总结一下你的需求：
  · 玩具：震动棒 + 电击器二代 + 灌肠机
  · 时长：约 5 分钟
  · 强度：渐进式，前戏温和→中间起伏→最后高潮
  · 节奏：前期舒缓，后期密集
  · 配合：分三阶段推进

生成的玩法方案：🔥【潮汐三重奏】

⏱ 总时长：约 4 分 30 秒

① 0-30s    前戏铺垫
            震动棒轻度震动 (A马达15%)
            电击器 A通道 模式1 强度20
② 30-90s   渐进升温
            震动棒逐步加强 (A30%→50%)
            电击器 B通道加入 模式3 强度40
③ 90-150s  灌肠介入
            灌肠机注水 40秒
            震动棒中速 (A60% C30%)
            电击器AB同步 模式5 强度80
④ 150-240s 高潮积累
            震动棒高速 (A80% C60%)
            电击器自定义 频率40Hz 强度150
⑤ 240-270s ⛰️ 高潮爆发
            震动棒满速 (A100% C100%)
            电击器 模式8 强度250
⑥ 270s     收尾
            全部停止

你觉得怎么样？可以：
  ① 直接定稿
  ② 调整某个参数（强度/时长/时机）
  ③ 重新设计
```

#### Phase 5：迭代

用户可以说：
```
"太强了，整体降 30%"
"灌肠机提前到一开始就注水"
"加一段循环，中段重复两次"
"电击器不用固定模式，改成自定义脉冲"
```

Agent 根据修改要求，不用重新问全部问题，**只调整对应的参数**，重新生成方案。

## 4. 对话原则

### 4.1 提问规范

```
1. 一次只问一个问题（最多附带一个子问题）
2. 每个问题提供推荐选项（3-4个）+ 开放式回答入口
3. 根据用户历史答案动态调整后续问题
4. 不要在用户还没回答时一下子问一堆
5. 无聊/显而易见的选项不要问（比如只有一个玩具时别问"你想用哪几个"）
```

### 4.2 推荐格式

每个问题遵循：

```
[问题]
→ 推荐：XXX
  A: 选项1
  B: 选项2  
  C: 选项3
也可以直接描述你想要的
```

### 4.3 迭代规范

```
- 用户说"调整"时，只修改对应的参数，不要全部重新生成
- 用户说"重来"时，回到 Phase 0 重新开始
- 迭代不超过 5 轮，超过则建议用户"基础定型后再微调"
- 每次迭代后显示对比（改了什么 + 新方案 vs 旧方案）
```

## 5. 输出格式（定稿时）

最终定稿时输出格式统一为：

```
🔥【方案名称】

--- Lua 脚本 ---
[Lua 代码]

--- 步骤解读 ---
⏱ 总时长：约 XX 秒
[步骤列表]

--- 玩法 ID ---
playbook_20260610_001
```

其中 `玩法 ID` 是方便 APP 缓存和索引用的唯一标识。

## 6. 测试场景

### 场景 1：全流程体验

```
已连接：震动棒 + 电击器二代 + 灌肠机
用户初始只说："随便来一套"
→ Agent 应主动引导，逐步问出：
  - 时长：3分钟
  - 强度：渐进
  - 节奏：前缓后紧
  - 最终生成完整方案
```

### 场景 2：用户一上来就很明确

```
已连接：电击器二代
用户说："电击器从弱到强，频率慢慢变化，4分钟"
→ Agent 应跳过不需要的基础问题，直接确认细节（固定模式还是自定义？频道分配？）
→ 生成方案
```

### 场景 3：迭代调整

```
方案定稿后，用户说："灌肠提前10秒"
→ Agent 不应该重新问所有问题，只调整 timing 参数
→ 重新展示方案
→ 再问"还有其他调整吗？"
```

### 场景 4：单玩具场景

```
已连接：震动棒
用户说："来个简单点的"
→ Agent 跳过玩具选择、协调问题
→ 只问时长、强度曲线、模式偏好
→ 生成简单方案
```
